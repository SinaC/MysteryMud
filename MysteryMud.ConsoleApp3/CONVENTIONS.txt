1. Core Naming Philosophy
Use role-first naming:

What it does → Systems
What it is → Components
What it marks → Tags
What it provides globally → Resources

Keep everything:

Singular (mostly)
Explicit
Suffix-based for clarity

2. Systems Naming
Convention:
[Domain][Action]System
Examples:
CombatDamageSystem
CombatAggroSystem
HealthRegenSystem
MovementSystem
NpcBehaviorSystem
TelnetInputSystem
Tips:
Group by domain prefix (Combat, Movement, AI, Network)
Avoid vague names like Manager or Handler
One system = one responsibility

3. Components Naming
Convention:
[Thing/Concept]Component

(or drop suffix if your ECS framework already enforces it)

Examples:
HealthComponent
PositionComponent
AggroComponent
DamageComponent
ConnectionComponent
InventoryComponent
Value rule:

Components = pure data only
No logic, ever.

4. Tags Naming
Tags = empty components used as flags

Convention:
[State/Marker]Tag
Examples:
DeadTag
PlayerTag
NpcTag
InCombatTag
AggressiveTag
LootableTag
Tip:

Keep tags:

Boolean in meaning
Stateless (no fields)

5. Resources (Global State)
Convention:
[Concept]Resource
Examples:
WorldTimeResource
GameConfigResource
AggroSettingsResource
SpawnTableResource
FormulaConfigResource

6. Formula / Calculation Layer
You mentioned heal/damage/aggro logic — good instinct separating it.

Convention:
[Domain]Calculator

or

[Domain]Service
Examples:
DamageCalculator
HealingCalculator
AggroCalculator
Alternative (more ECS-pure):

Move orchestration to systems:

DamageCalculationSystem
AggroUpdateSystem

Use calculators as:

Stateless helpers
Pure functions

7. Factories
Convention:
[EntityType]Factory
Examples:
PlayerFactory
NpcFactory
ItemFactory
Tip:

Factories should:

Assemble entities with components
Not contain game logic

8. Networking / Telnet
Convention:
[Domain][Role]
Examples:
TelnetServer
TelnetSession
TelnetConnection
CommandParser
InputBuffer

10. Multi-Project Structure (Recommended)
Split into assemblies if your project grows:

Game.Core
Game.ECS
Game.Systems
Game.Simulation
Game.Networking
Game.Persistence
Game.Content
Responsibilities:
Core → base abstractions
ECS → entity/component/system framework
Systems → gameplay logic
Simulation → calculators & rules
Networking → telnet, protocol
Persistence → save/load
Content → data definitions

11. Consistency Rules (Most Important)
If you follow nothing else, follow these:

Every system ends with System
Every tag ends with Tag
Every resource ends with Resource
No logic inside components
Systems don’t store state (use resources instead)
Domain prefix always comes first
